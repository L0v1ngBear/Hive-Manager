# P0-20260714-001 Hive 发布链路事故

## 事故信息

- 事故编号：P0-20260714-001
- 严重等级：P0
- 当前状态：处理中
- 发生时间：2026-07-14 发布窗口
- 影响范围：Hive 线上发布、数据库迁移、管理端与小程序后端启动、HTTPS 入口
- 用户影响：发布被阻断；后端容器恢复阶段存在接口 502 和服务不可用风险，实际用户影响仍需结合线上访问日志确认
- 恢复标准：完整发布验收通过，管理端和小程序后端接口无 404/502，迁移记录全部成功，公网证书有效期超过 30 天，并持续观察 24 小时

## 事故摘要

本次发布连续触发多个发布门禁和恢复失败：管理端静态目录残留旧构建、TLS 证书临近到期、权限 V3 迁移存在混合排序规则比较、权限关联未完全收敛、线上结构验证脚本错误解释反引号，以及部署包中的旧证书覆盖服务器证书。迁移失败后发布脚本尝试恢复容器，但恢复过程未完整结束，扩大了服务不可用风险。

该事件按 P0 处理，不允许通过跳过门禁、修改成功迁移历史或忽略证书检查的方式上线。

## 事件链路

1. 管理端发布采用覆盖上传，旧的带哈希 JavaScript 文件未被删除，信息渠道契约门禁检测到遗留的 `deliveryDate/delivery_date`。
2. 网页端静态目录完成干净替换后，TLS 门禁检测到证书将在 30 天内到期。
3. 权限 V3 迁移在临时权限目录与现有权限表比较时触发 `utf8mb4_0900_ai_ci` 与 `utf8mb4_general_ci` 冲突。
4. 迁移失败后后端容器恢复未完整完成，出现启动失败和 502 风险。
5. 权限 V3 主迁移成功后，结构验收发现无效角色权限关系仍然存在。
6. 新增权限关系收敛迁移后，验证脚本在双引号 SQL 中使用未转义的 \`user\`，被 Bash 当作命令替换，产生 `Table 'hive.u' doesn't exist`。
7. 后续完整目录上传又将本地旧证书覆盖到服务器，TLS 门禁再次阻断发布。

## 根因

### RC1：发布目录与运行时数据未彻底隔离

代码、静态资源、证书和运行时目录共用同一部署树，直接覆盖上传会污染或覆盖线上状态。网页静态文件需要删除旧哈希文件，TLS 证书则必须保留服务器版本，两者需要相反的同步策略，但原流程没有强制区分。

### RC2：发布不是基于暂存目录的原子替换

管理端 `dist` 使用合并覆盖方式上传，没有 `rsync --delete`、文件清单或原子目录切换，导致旧构建继续被发布门禁扫描。

### RC3：数据库迁移缺少真实排序规则矩阵验证

权限 V3 SQL 只按单一数据库排序规则设计，没有覆盖现有数据库中混用 `utf8mb4_0900_ai_ci` 和 `utf8mb4_general_ci` 的情况。

### RC4：Shell 内嵌 SQL 缺少转义检查

验证脚本在 Bash 双引号字符串中直接使用 SQL 反引号，触发命令替换。发布前静态检查没有覆盖这一类 Shell/SQL 边界错误。

### RC5：失败恢复链路缺少强制完成和二次验证

迁移失败后的容器恢复依赖同一 Compose 操作，基础服务等待或后端启动异常会导致恢复动作取消，没有独立验证恢复后的 JAR、端口和 HTTP 状态。

## 已完成修复

- 管理端生产构建改为干净同步，发布门禁能够输出遗留字段对应的具体文件。
- 权限 V3 跨表权限编码和角色编码比较改为二进制精确比较。
- 新增 `V20260714_001_permission_relation_convergence.sql`，软删除无效角色权限和用户权限覆盖关系。
- 修复 `verify-online-schema.sh` 中 `user` 表反引号的 Bash 转义。
- 本地部署包已移除 TLS 证书和私钥，上传文档明确要求保留服务器 `nginx/certs/` 与 `nginx/letsencrypt/`。
- 两个后端镜像启动后均执行容器内 JAR 与部署包 JAR 的 SHA-256 比对。
- 发布继续保留迁移校验、TLS 有效期、在线冒烟和失败状态记录门禁。
