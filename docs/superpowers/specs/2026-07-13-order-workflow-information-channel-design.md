# 订单流转与信息渠道统一改造设计

## 目标

本次改造统一管理端、小程序端和两个后端的订单契约，解决图纸预算流转、流转前无法补充订单信息、取消无原因、审核人配置不一致、后生成流转二维码错误，以及“交付日期”已经不符合实际业务含义的问题。

用户已确认采用直接替换方案：数据库、Java 模型、接口 JSON 和前端状态统一使用 `informationChannel` / `information_channel`，不保留 `deliveryDate` / `delivery_date` 兼容层。

## 全局约束

1. 订单仍是一个统一入口，销售和生产人员通过状态权限及数据范围看到各自订单，不恢复销售订单、生产订单两个页面。
2. 管理端和小程序端使用相同字段名、状态码、状态文案和流转规则。
3. 所有状态变更、审核创建和二维码流转都由后端验证权限、租户、当前状态及目标状态，前端提示不能代替后端校验。
4. 不修改任何已经执行过的历史迁移 SQL。新增迁移必须追加到部署包迁移清单。
5. 新版本是非兼容发布，数据库迁移、两个后端、管理网页和小程序必须作为同一个发布批次上线，不允许长期混跑旧接口。
6. 每个代码切片先写失败测试，再实现最小修复；合并后执行跨仓库契约检查和打包验证。

## 一、订单状态机

### 普通订单

普通订单继续使用现有统一状态机和状态权限。点击下一步时，客户端不直接流转，而是进入“流转编辑”意图：

1. 加载最新订单详情和版本信息。
2. 打开现有订单编辑表单。
3. 用户补全下一阶段所需信息。
4. 先以当前旧状态保存字段，不在保存请求中预写目标状态。
5. 保存成功后再调用下一步接口。
6. 下一步失败时保留已保存字段和旧状态，并提示“订单已保存，流转未完成，请重试”。

当目标状态为 `shipped` 时，表单必须显示并校验物流公司和物流单号。`pending_ship -> shipped` 必须提交订单审批：推进请求先保存物流及本次编辑信息，订单状态保持 `pending_ship`；全部审核人通过后才变为 `shipped`，任一拒绝则保持 `pending_ship`。后端下一步接口仍须再次校验物流信息、重复待审和审核人范围，防止绕过客户端。

### 图纸预算订单

图纸预算订单只允许：

```text
budgeting -> budget_completed
```

`budget_completed` 是正式终态，中文显示“预算已完成”。终态再次扫码或点击下一步返回明确的业务错误，不得进入普通订单的 `pending_confirm`。管理端和小程序端均提供独立、可点击的“预算已完成”筛选项，横向拖动筛选栏只滚动，只有点击才改变选中项。

## 二、信息渠道直接替换

### 统一契约

新字段定义：

```text
Java / JSON: informationChannel: String
MySQL: information_channel VARCHAR(100)
界面标签: 信息渠道
输入方式: 普通文本框
```

覆盖对象包括：

- 销售订单实体、创建、编辑、列表、详情、导出和打印；
- 生产订单实体、创建、编辑、列表、详情和销售转生产同步；
- 安装任务实体、列表、详情和订单完成后的同步；
- 小程序订单列表、详情、创建和编辑；
- 管理端订单列表、详情、创建、编辑和安装任务页面；
- 手册、接口契约测试和打印模板数据。

普通销售订单和生产订单的信息渠道必填，`drawing_budget` 可为空，保持原有分类的必填边界。字段不做日期解析、未来日期校验、日期格式化、日期排序或日期范围筛选。

### 数据迁移

新增版本化迁移，不修改历史文件。迁移执行以下操作：

1. 为 `sales_order`、`production_order`、`installation_task` 新增 `information_channel VARCHAR(100)`。
2. 将历史非空日期转换为 `历史交付日期：YYYY-MM-DD`；无法标准化的历史文本使用 `历史交付日期：<原值>`。
3. 删除依赖 `delivery_date` 的索引。
4. 删除三个表的 `delivery_date` 列。
5. 为 `sales_order` 新增 `cancel_reason VARCHAR(500)`。
6. 将迁移追加到 `db-migrations/migration_manifest.txt`，并更新在线结构验证和迁移不可变测试。

自动迁移执行前必须有可验证备份。由于接口不兼容，失败回滚以数据库备份和上一整套发布快照为单位，不做单个服务回退。

## 三、取消原因

取消订单使用独立字段 `cancelReason` / `cancel_reason`，不复用订单备注。

1. 用户选择取消时先弹出原因输入，去除首尾空白后不能为空，最大 500 字。
2. 后端在写状态、建审核记录或写日志之前验证原因。
3. 订单详情、取消审核详情和状态日志均显示取消原因。
4. 状态日志备注使用 `取消原因：<内容>`，审核对象同时保留结构化 `cancelReason`。
5. 取消审核被拒绝时保留原因和审核轨迹，订单回到原业务状态。

## 四、审核人选择和审核中心

创建、编辑、流转和回退需要审核时，统一使用复选框式审核人选择框，提交 `auditorIds: number[]`。显式选择优先于默认审核人；未显式选择时由后端读取对应审核类型的默认审核人。

默认审核人配置按审核类型授权，不再统一依赖 `approval:finance:audit`。被指派的审核人可以查看并处理自己待审的记录，不要求同时拥有宽泛的 `order:list` 或 `order:detail`，但不能借此查看其他订单。

多审核人规则保持：全部通过才通过，任一拒绝则结束其余待审。管理端和小程序端统一最多选择 30 人，并在后端校验租户、在职状态、审核权限和重复 ID。

订单审批至少覆盖两个前进场景：`pending_pay -> pending_material` 和 `pending_ship -> shipped`。审批记录必须保存明确目标状态，审核中心不能只根据固定旧状态猜测目标；发货审批提交时物流信息已经落库，但审核完成前订单仍属于待发货范围。

部署迁移必须补齐 `approval_default_auditor`、`approval_auditor_candidate` 所需表和索引；已有表时安全跳过，不修改历史迁移。

## 五、订单流转二维码

二维码原始内容固定为：

```text
HIVE_ORDER_FLOW:<sales|production>:<base64url-hmac-sha256>:<orderId>
```

签名生成、打印模板、补打和小程序解析必须使用同一协议。模板渲染只执行一次占位符替换，不能把已经插入的 JSON 花括号再次识别为模板变量。TSPL 输出要正确转义引号和控制字符，自动打印与补打走同一渲染函数。

验收除单元测试外，还要打印 40mm 实物二维码并由小程序扫码，确认扫描值和后端签名验证一致。

## 六、小程序流转编辑

小程序复用 `salesOrderCreate` 页面并增加编辑模式，不新增第二套重复表单。订单列表点击下一步时传入订单 ID 和 `advance=1`：

1. 编辑页拉取详情并显示当前状态。
2. 用户保存后调用更新接口，更新请求保持当前状态。
3. 更新成功后调用 `flow-advance` 或统一下一步接口。
4. 成功后返回订单页并刷新；更新成功但流转失败时留在编辑页并显示可重试提示。
5. 无编辑权限或无目标状态权限时，入口置灰且内容不可见；后端仍做最终鉴权。

## 七、接口与错误语义

请求和响应不再出现 `deliveryDate`。服务端遇到旧字段不能静默接受，避免旧客户端继续写入错误语义。

关键错误使用明确业务信息：

- 信息渠道缺失：`信息渠道不能为空`；
- 图纸预算终态：`预算已完成，不能继续流转`；
- 取消原因缺失：`取消原因不能为空`；
- 物流信息缺失：`流转到已发货前请填写物流公司和物流单号`；
- 发货审批提交成功：客户端提示 `已提交发货审批，审批通过后进入已发货`；
- 保存成功但客户端下一步失败：客户端提示 `订单已保存，流转未完成，请重试`；
- 审核人无效：指出无效、离职、跨租户或无审核权限的审核人。

## 八、测试与发布验收

### 未开票时间预警

订单页复用现有预警设置和缓存体系，新增 `invoiceWarningDays`，默认 30 天，可配置范围 1 至 365 天。预警从订单 `create_time` 起算，后续编辑不能重置；满足 `is_invoice = 0`、超过阈值且不属于 `drawing_budget`、`pending_cancel`、`cancelled` 的订单进入未开票预警。

管理网页和小程序订单页统一提供“未开票预警”统计卡、仅看预警筛选及行内“已未开票 N 天”标识。分页接口返回 `invoiceWarning` 和 `uninvoicedDays`，统计接口返回 `invoice_warning`；服务端按当前用户可见状态范围统计，不能泄露无权订单。订单改为已开票后立即失效相关缓存并退出预警。

### 自动测试

- 管理后端：DTO/实体/VO 契约、状态机、待收款与待发货审批、取消原因、审核范围、同步安装任务、打印载荷。
- 小程序后端：统一订单接口、图纸预算终态、待发货审批、权限范围、信息渠道、二维码签名、安装同步。
- 管理网页：文本输入及 payload、流转编辑顺序、物流校验、预算已完成筛选、未开票预警、取消原因、审核人选择。
- 小程序前端：创建/编辑 payload、列表详情展示、流转编辑顺序、横向筛选、未开票预警、扫码解析、权限禁用。
- 部署包：新迁移存在且在清单末尾、历史迁移哈希不变、三表旧列消失、新列存在、后端 jar 与容器哈希一致。

### 发布顺序

1. 停止业务写入并完成数据库备份验证。
2. 执行新增迁移并运行结构校验。
3. 部署管理端后端和小程序后端新 jar，确认容器内外 SHA256 一致。
4. 部署管理网页 dist。
5. 上传并发布小程序代码包。
6. 执行订单创建、编辑后流转、取消审核、图纸预算终态、安装任务同步、二维码实扫和权限范围验收。
