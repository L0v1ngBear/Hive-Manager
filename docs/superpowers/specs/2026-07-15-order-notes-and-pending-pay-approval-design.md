# 订单多条备注与待收款审批触发修复设计

日期：2026-07-15

## 1. 目标

本次改造包含两个关联但边界独立的目标：

1. 将订单业务备注从 `sales_order.remark` 单个大文本字段改为独立的多条备注记录，网页端和小程序端保持一致。
2. 修复创建待收款订单时错误生成审批候选的问题。只有用户主动执行 `待收款 -> 备料中` 状态推进时，系统才发起物料审批。

正式上线会清理原测试数据，因此本次不迁移、不兼容旧订单备注数据，也不保留旧权限码和旧接口分支。

## 2. 范围边界

### 2.1 纳入范围

- 网页端订单新建、编辑和详情。
- 小程序端订单新建、编辑和详情。
- 管理端后端和小程序后端的订单备注数据契约。
- 数据库迁移、权限目录、内置角色权限和用户独立权限。
- 订单操作日志中的备注新增、修改行为记录。
- 待收款订单创建、推进、审批中心候选生成和审核权限。

### 2.2 不纳入范围

- 状态流转日志中的审批意见、操作说明和取消说明仍是操作日志，不转换为订单业务备注。
- 安装任务的施工备注、特殊及异常情况说明保持独立。
- 不提供订单备注删除能力。
- 不恢复生产订单和销售订单旧页面或旧权限兼容。

## 3. 订单备注数据模型

新增表 `sales_order_note`：

| 字段 | 类型 | 规则 |
| --- | --- | --- |
| `id` | bigint | 主键 |
| `tenant_code` | varchar(64) | 必填，租户隔离 |
| `order_id` | varchar(64) | 必填，关联统一订单 |
| `content` | varchar(1000) | 必填，去除首尾空格后不可为空 |
| `creator_user_id` | bigint | 必填，由登录上下文生成 |
| `creator_name` | varchar(100) | 必填，保存创建时显示名 |
| `updater_user_id` | bigint | 必填，由登录上下文生成 |
| `updater_name` | varchar(100) | 必填，保存最后修改人显示名 |
| `version` | int | 必填，默认 0，用于乐观锁 |
| `create_time` | datetime | 必填，数据库或服务端生成 |
| `update_time` | datetime | 必填，新增和修改时更新 |

索引：

- `(tenant_code, order_id, update_time, id)`：订单详情查询和倒序展示。
- 不设置软删除字段，不提供删除 SQL 和删除接口。

新迁移直接删除 `sales_order.remark` 业务字段。历史迁移文件保持不可变，只新增更高版本迁移并追加到 `migration_manifest.txt`。

## 4. 前后端数据契约

订单新建和编辑请求使用：

```json
{
  "notes": [
    {
      "id": null,
      "content": "客户要求周五前确认包装",
      "version": null
    },
    {
      "id": 1024,
      "content": "颜色样已于今日确认",
      "version": 2
    }
  ]
}
```

规则：

- `id` 为空表示新增。
- `id` 存在表示修改现有备注；只有内容变化时才更新修改人、修改时间和版本号。
- 请求未携带的已有备注保持不变，不解释为删除。
- 备注 ID 必须属于当前租户和当前订单，否则拒绝整个请求。
- 每单最多 50 条备注，单条最多 1000 字。
- 新建订单和备注必须在同一数据库事务中提交。
- 订单详情直接返回完整 `notes`，按 `update_time DESC, id DESC` 排序。
- 删除所有订单业务 DTO、VO、实体和映射中的旧 `remark` 字段；状态流转请求中的操作说明保留，但命名和业务备注分离。

当版本号不一致时返回明确的并发修改错误，前端提示“备注已被其他人修改，请刷新后重试”，不得静默覆盖。

## 5. 页面交互

网页端和小程序端统一采用“备注明细”列表：

- 原大文本框彻底移除。
- 交互与商品明细一致，提供“新增备注”按钮，可连续增加多行。
- 新增但尚未保存的行显示“保存后记录修改时间”。
- 已保存备注显示内容、最后修改时间、最后修改人和编辑入口。
- 已保存备注不显示删除按钮，接口层也不接受删除；尚未保存的新备注行允许撤销。
- 只有内容实际变化时才显示新的修改时间。
- 订单详情默认最新修改的备注在前。
- 订单列表当前的物流单号列保持不变，不重新加入备注列。

## 6. 权限设计

新增 Permission Catalog V3 精确叶子权限：

- `order:note:view`
- `order:note:create`
- `order:note:update`
- `order:audit:material`

权限规则：

- 查看备注还必须满足对应订单状态的查看权限和订单数据范围。
- 新增、修改备注还必须满足订单数据范围；不因持有备注权限而扩大可见订单范围。
- 内置角色按其现有订单职责同步授予对应备注权限。
- 用户独立允许/禁止权限继续优先于角色权限。
- `order:audit:material` 专用于待收款转备料审批。
- 禁止复用 `order:audit:shipment`，彻底移除已停用的 `approval:order:audit` 调用。

## 7. 操作日志

新增或修改备注时写入订单操作记录：

- 操作类型：`NOTE_CREATE` 或 `NOTE_UPDATE`。
- 记录订单号、备注 ID、操作人和操作时间。
- 操作日志不复制完整备注内容，业务内容只保存在 `sales_order_note`。
- 备注自身保存创建人、最后修改人、创建时间和最后修改时间。

状态流转日志继续记录审批意见和状态变化，两类记录不混用。

## 8. 待收款审批触发修复

### 8.1 当前根因

管理端后端在创建订单后，只要订单状态是 `pending_pay` 就同步审批候选，把“订单当前状态”错误等同于“已申请推进到备料中”。同时两套后端分别复用了发货审批权限和已停用旧权限码。

### 8.2 正确流程

1. 创建状态为 `pending_pay` 的订单：只保存订单，不创建审批候选，不进入审批中心待办。
2. 用户点击推进并提交 `pending_pay -> pending_material`：保存必要的订单编辑内容后，创建 `ORDER` 类型审批候选和 `approval_pending` 操作记录。
3. 审批通过：订单进入 `pending_material`。
4. 审批拒绝：订单保持 `pending_pay`，记录拒绝意见。

审批业务编号继续使用 `sales:<orderId>`。审核人解析顺序保持为：显式审核人、租户默认审核人、首个拥有 `order:audit:material` 且不是申请人的有效员工。

审批中心只查询存在有效审批候选或明确待审批操作记录的订单，不再按 `pending_pay` 状态全量推导待办。GET 查询不得自动补建审批候选或产生数据库写入。

## 9. 接口收口

统一使用规范订单接口：

- `POST /orders`
- `PUT /orders/{id}`
- `GET /orders/{id}`
- `POST /orders/{id}/advance`

网页端和小程序端使用同一数据契约。旧 `/order/create`、`/order/next/{id}` 分支不继续兼容。

## 10. 错误处理

- 空备注：400，提示“备注内容不能为空”。
- 超过 1000 字：400，提示“单条备注不能超过1000字”。
- 超过 50 条：400，提示“每个订单最多添加50条备注”。
- 跨租户或跨订单备注 ID：拒绝请求并记录安全事件。
- 乐观锁冲突：409，提示刷新后重试。
- 无权限：403；前端置灰入口，不加载不可见备注内容。
- 审核人为空：推进申请失败，订单仍保持待收款，不创建半成品候选记录。

## 11. 测试与验收

### 11.1 后端

- 新建订单同时保存 0 条、1 条和多条备注。
- 修改一条备注只更新该条的修改人、修改时间和版本。
- 省略已有备注不会删除记录。
- 删除请求不存在或被拒绝。
- 跨租户、跨订单 ID、空内容、超长和超数量均被拒绝。
- 并发修改返回 409。
- 创建待收款订单不产生审批候选。
- 只有待收款推进到备料中才生成物料审批。
- 审批通过、拒绝、无审核人和重复申请均保持一致状态。
- 管理端后端和小程序后端契约测试一致。

### 11.2 前端

- 网页端和小程序端均可新增多条、修改已有备注；已保存备注无删除入口，未保存的新备注行可以撤销。
- 已保存备注显示最后修改时间和修改人。
- 无备注权限时入口置灰且内容不可见。
- 乐观锁冲突提示明确，不覆盖他人修改。
- 原大文本框和旧 `remark` 请求字段不存在。

### 11.3 发布门禁

- 新迁移已追加到清单，历史 SQL 哈希不变。
- 前端、两个后端 JAR 和部署包同时重建。
- 静态扫描确保订单业务契约不再引用旧 `sales_order.remark`。
- Permission Catalog、网页端、小程序端和两个后端的四个新权限码完全一致。
