# Hive 系统内置角色与审批权限重建设计

日期：2026-07-10

## 背景与根因

当前权限不足并非单一账号配置错误，而是以下问题叠加：

1. 小程序审批中心默认进入订单审批，并在权限汇总返回前并发请求 `/approval/order/list`。没有 `order:list` 的员工进入页面就会收到 403。
2. 新版小程序依赖 `/approval/summary` 的新建能力字段；旧部署包不返回这些字段时，前端按无权限处理。当前部署包后端 JAR 早于本次能力字段代码。
3. 历史内置角色由多轮 SQL 迁移叠加形成。专业岗位角色没有统一包含普通员工的考勤、请假、财务、离职等自助权限，员工从 `EMPLOYEE` 换成专业角色后可能丢失基础能力。
4. 新租户运行时只保证 `ADMIN` 和 `EMPLOYEE` 两个角色存在，历史 SQL 创建的其他内置角色不会自动出现在迁移之后创建的租户中。
5. 管理端、小程序端的默认员工权限清单不一致。小程序回退逻辑还包含订单、库存、质量、客户等过宽权限。
6. 已删除的 AI 建议功能仍在历史迁移中留下 `AI_MANAGER` 和 `dashboard:ai*` 权限；`TENANT_OWNER` 与当前代码使用的 `ADMIN` 负责人角色功能重复。
7. 安装任务暂时复用 `order:list` 控制读写，没有独立权限，无法给安装人员最小授权。

## 目标

- 形成一套明确、可解释、可重复创建的系统内置角色矩阵。
- 所有岗位角色统一拥有员工基础自助权限，不因换岗丢失请假、考勤等能力。
- 专员和负责人分层，负责人拥有本部门配置、审核和导出能力，专员只拥有日常操作能力。
- 订单继续按状态权限分工，不恢复销售订单、生产订单两套概念。
- 安装任务使用独立权限，不再借用订单列表权限控制全部操作。
- 小程序审批中心只请求当前员工有权访问的数据；无权限页签置灰且不展示内容。
- 普通员工可查看自己发起的审批，但不能查看其他员工的审批。
- 员工个人 `GRANT`/`DENY` 权限继续覆盖角色权限，`DENY` 优先。
- 历史迁移文件保持不变，只追加新迁移。

## 方案比较

### 方案一：岗位双层角色加统一基础权限（采用）

保留业务岗位的“专员/负责人”两层，为所有非管理员内置角色物化同一份员工基础权限，再叠加岗位权限。该方案在可管理性和最小授权之间平衡最好，也与现有角色配置和个人权限覆盖机制一致。

### 方案二：每个部门只保留一个角色

角色数量少，但日常操作和审核配置只能依赖大量个人权限覆盖。企业人数增加后，个人差异难以审计，也容易出现离职或调岗后残留权限。

### 方案三：仅修补当前历史角色

改动最小，但会继续保留重复角色、AI 残留、新租户缺角色、安装任务借用订单权限等结构问题，后续仍需重复修补。

新版保留 20 个租户内置角色：

| `EMPLOYEE` | 普通员工 | 考勤、本人审批、公告和文档只读 |
| `SALES_STAFF` | 销售专员 | 客户、报价、订单创建和前段状态处理 |

| `HR_STAFF` | 人事专员 | 员工只读、考勤记录、请假和离职列表 |

| `EQUIPMENT_MANAGER` | 设备负责人 | 设备巡检员能力加设备维护、巡检记录和导出 |

以下历史角色退役：

- `TENANT_OWNER`：用户绑定迁移到 `ADMIN` 后软删除。
- `AI_MANAGER`：用户绑定迁移到 `EMPLOYEE` 后软删除。

平台角色 `SUPER_ADMIN`、`PLATFORM_ADMIN` 不属于租户内置角色，不参与本次重置。

## 权限组合规则

### 员工基础权限

除 `ADMIN` 外，全部内置角色都物化以下基础权限：

- `attendance:punch`
- `attendance:record:list`
- `approval:leave:submit`
- `approval:leave:detail`
- `approval:finance:submit`
- `approval:finance:detail`
- `approval:resignation:submit`
- `approval:resignation:detail`
- `document:list`
- `document:breadcrumbs`
- `notification:announcement:list`

基础权限不包含订单、库存、客户、质量、价格、标签或员工管理，避免普通员工默认获得业务数据权限。

### 专员与负责人

- 专员角色 = 员工基础权限 + 本岗位日常操作权限。
- 负责人角色 = 对应专员权限 + 本岗位配置、审核、撤销、发布和导出权限。
- 负责人不自动获得其他部门权限。
- `APPROVAL_MANAGER` 只增加审核所需的列表、详情和审核权限，不增加员工维护、角色配置或直接业务数据修改权限。

### 订单状态权限

继续使用统一订单权限：

- 基础入口：`order:list`、`order:detail`、`order:create`。
- 状态责任：`order:status:<status>`。
- 销售负责预算、待确认和取消相关状态；财务负责待收款；仓储负责备料、待发货和发货；生产负责备料、生产和待发货；安装负责已发货和已完成的交付查看。
- 只有企业负责人获得全部订单状态权限；销售负责人也仅能查看预算、待确认、待取消和已取消等销售职责阶段，不能查看备料、生产和待发货阶段。
- 生产专员和生产负责人仅能查看备料、生产中和待发货阶段；`order:list` 只开放统一订单入口，不能单独放开任何订单数据。
- 旧 `sales:order:*`、`production:order:*` 不再写入新版内置角色矩阵。

### 安装任务权限

新增权限节点：

- `installation:*`
- `installation:list`
- `installation:update`
- `installation:attachment:upload`
- `installation:attachment:download`

安装任务接口分别使用对应权限，不再统一依赖 `order:list`。安装角色可获得所需订单交付状态的查看能力，但不能因此修改订单其他阶段。

### 管理权限

`ADMIN` 获得所有未删除的租户业务权限，但排除：

- `*`、`*:*`、`super`、`developer:super`
- `platform` 及 `platform:*`
- 已退役的 `dashboard:ai` 及其子权限

角色管理、员工管理和个人权限覆盖配置只默认授予 `ADMIN`。企业可通过自定义角色或个人覆盖再委派。

## 审批中心访问设计

### 后端能力摘要

`GET /approval/summary` 除已有新建能力外，增加：

- `canViewOrder`
- `canViewQuality`
- `canReviewFinance`
- `canReviewLeave`
- `canReviewResignation`

财务、请假和离职页签的可访问条件为“可新建或可审核”；订单和质量页签必须拥有对应查看/处理权限。

### 查询范围授权

财务、请假和离职列表按 `scope` 动态授权：

- `mine`：拥有对应提交权限、列表权限或审核权限即可；服务层强制按当前用户 `applyUserId` 过滤。
- `pending`：必须拥有对应列表/审核权限，并由服务层按当前审核人过滤。
- `all`：必须拥有对应列表权限。
- 非法或空 `scope` 归一化为 `pending`，不能借此绕过授权。

详情读取增加行级校验：申请人可查看自己的记录，当前审核人可查看分配给自己的记录，拥有对应全量列表/详情权限的负责人可查看其他记录。仅知道业务编号不能越权读取他人审批。

### 小程序行为

1. 页面先请求权限摘要，不再并发请求默认订单列表。
2. 根据摘要选择第一个可访问页签；没有订单权限时不会请求 `/approval/order/list`。
3. 无权页签保留可见但置灰，点击只提示权限不足，列表内容不加载。
4. 只有新建权限、没有审核权限时，页签固定使用“我发起的”，不显示“待我处理”和“全部”。
5. 有审核权限时显示现有范围筛选。
6. 摘要请求失败时显示“权限信息加载失败”，不把部署或网络错误伪装成账号无权限。

## 数据迁移策略

新增版本化迁移 `V20260710_003_builtin_role_permission_matrix.sql`，并通过后续迁移 `V20260710_004_order_role_status_scope.sql` 固化销售/生产阶段隔离，不修改任何历史迁移。

迁移按以下顺序执行：

1. 创建缺失的安装任务和审批订单审核权限节点。
2. 对所有有效租户创建或恢复 20 个内置角色，并统一名称和 `is_system=1`。
3. 将 `TENANT_OWNER` 的有效用户绑定幂等迁移到 `ADMIN`。
4. 将 `AI_MANAGER` 的有效用户绑定幂等迁移到 `EMPLOYEE`。
5. 软删除退役角色的用户绑定、角色权限和角色记录。
6. 仅重置本次保留的内置角色权限关系，再按新版矩阵写入。
7. 软删除 `dashboard:ai` 权限及其角色关系和个人覆盖关系。
8. 保留全部自定义角色、保留角色用户绑定、保留与有效权限相关的个人 `GRANT`/`DENY`。

迁移必须可在同一数据库重复执行而不产生重复关系。部署脚本继续通过迁移清单和 SHA-256 历史校验执行，新文件加入清单和静态健康检查。

## 新租户与运行时一致性

管理端增加集中的内置角色目录和角色初始化服务：

- 新租户创建或负责人账号初始化时，创建全部 20 个内置角色及默认权限。
- 运行时初始化只处理缺失角色和首次创建的权限，不在每次登录或启动时覆盖管理员后续手工调整。
- `EMPLOYEE` 的基础权限由同一目录提供给管理端加入组织流程。
- 小程序加入组织的回退权限清单同步为同一份最小基础权限，删除订单、库存、客户、质量和标签等过宽默认权限。
- 静态测试校验管理端目录、小程序基础清单和迁移 SQL 中的角色代码与基础权限一致。

## 个人权限覆盖

现有 `sys_user_permission` 规则保持不变：

- `GRANT` 在角色权限之外增加个人权限。
- `DENY` 以 `!perm_code` 进入权限上下文，并优先于角色权限和通配权限。
- 角色矩阵迁移不删除有效业务权限的个人覆盖。
- 角色或个人权限保存后同时清理管理端和小程序权限缓存。

为避免数据库迁移完成后继续读取旧 Redis 权限集合，管理端和小程序权限缓存键升级到新版本命名；旧键自然过期，不使用全库 `FLUSHDB`。

## 代码与部署边界

管理端后端：

- 内置角色目录与新租户初始化服务。
- 安装任务独立权限常量和接口注解。
- 审批范围动态授权及详情行级校验。
- 权限缓存键版本升级。
- 角色矩阵与安全边界测试。

小程序后端：

- 审批摘要访问能力字段。
- 审批范围动态授权及详情行级校验。
- 最小员工基础权限同步。
- 权限缓存键版本升级。
- 审批权限测试。

小程序前端：

- 审批摘要先行加载。
- 页签置灰、首个可访问页签选择和范围筛选约束。
- 明确区分“权限不足”和“权限信息加载失败”。

部署包：

- 新迁移 SQL 和迁移清单。
- 数据库预检/健康检查增加角色矩阵校验。
- 重新构建管理端后端 JAR、小程序后端 JAR 和小程序源码包。
- 部署顺序固定为：备份与迁移 -> 两个后端 -> 在线接口检查 -> 小程序发布。

管理端前端角色页面继续读取数据库权限树和角色权限 ID，现有勾选回显和个人权限抽屉不需要重写。

## 验收标准

1. 每个有效租户拥有 20 个新版内置角色，名称和权限矩阵一致。
2. `TENANT_OWNER`、`AI_MANAGER` 和 `dashboard:ai*` 不再作为有效角色或权限出现，相关用户获得安全替代角色。
3. 所有专业岗位角色均能打卡、查看自己的考勤、提交并查看自己的请假/财务/离职申请。
4. 普通员工不能查看其他员工的审批、订单、库存、客户、质量或员工数据。
5. 负责人只获得本部门管理能力；只有 `ADMIN` 默认拥有角色和员工权限配置能力。
6. 安装专员可以处理安装任务，但没有无关订单阶段和其他部门权限。
7. 小程序审批中心进入时不再因为默认订单页签弹出权限不足。
8. 无权页签置灰且不发请求；新建权限不足时表单不可见。
9. 角色权限与个人覆盖修改后，管理端和小程序在新请求中读取新权限，不等待旧缓存 30 分钟。
10. 自定义角色和有效个人覆盖在迁移后保持不变。
11. 历史迁移 SHA-256 不发生变化，新迁移可重复执行。
12. 管理端、小程序后端、小程序前端和部署静态/在线检查全部通过。
